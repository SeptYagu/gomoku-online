# 五子棋在线对弈网站搭建计划

更新日期：2026-06-23

## 1. 项目目标

做一个打开即玩的五子棋在线对弈网站，前期重点是“可玩、稳定、移动端舒服、容易分享”，中后期通过广告变现和自然搜索流量盈利。

核心目标：

- 免注册即可开始游戏。
- 支持人机、本地双人、好友房、随机匹配。
- 移动端和桌面端都能顺畅对弈。
- 广告放在自然间隙和低误触区域，避免影响棋盘操作。
- 通过规则页、教程页、排行榜、每日挑战提升留存和搜索流量。

- 账号系统不是第一天必须有，游客模式反而能降低流失。

## 3. 产品定位

第一版定位：

- 一个轻量、干净、适合反复玩的在线五子棋网站。
- 默认进入游戏界面，不做传统营销首页。
- 重点支持“我想马上下一盘”和“我想发链接给朋友下一盘”。

目标用户：

- 搜索“五子棋在线”“gomoku online”“five in a row online”的轻度玩家。
- 想和朋友远程下一盘的用户。
- 想练习棋力的人机玩家。
- 移动端碎片时间玩家。

## 4. 推荐技术方案

前端：

- Next.js + React + TypeScript。
- 棋盘使用 Canvas 或稳定 CSS Grid。若优先移动端丝滑体验，推荐 Canvas。
- 状态管理优先使用 React state/context，后续复杂后再引入 Zustand。

后端：

- Node.js + Fastify 或 Express。
- Socket.IO 处理实时房间、落子同步、断线重连。
- REST API 处理账号、排行榜、历史记录、内容页数据。

数据层：

- PostgreSQL 存用户、对局、排行榜、广告/实验配置。
- Redis 存在线状态、匹配队列、短期房间状态和限流数据。
- Prisma 作为 ORM。

部署：

- MVP 可先单体部署，减少复杂度。
- 推荐 Fly.io、Railway、Render 或 VPS 部署 Node 服务。
- 如果前后端拆开，前端可部署 Vercel/Cloudflare Pages，实时服务单独部署。

广告：

- 早期使用普通 AdSense 展示广告。
- 站点和流量稳定后申请 Google H5 Games Ads。
- 插屏或激励广告只放在对局结束、重新匹配前、等待匹配等自然间隙。

## 5. 阶段 0：项目初始化与原型确认

目标：

- 搭好工程骨架，确定视觉方向和最小可玩范围。

功能范围：

- 创建 Next.js/TypeScript 项目。
- 设置代码格式、Lint、测试框架。
- 确定棋盘尺寸、布局、移动端断点。
- 建立核心规则模块的接口设计。

技术任务：

- 初始化 `package.json`、Next.js、TypeScript、ESLint、Prettier。
- 建立目录结构：
  - `src/app`：页面和路由。
  - `src/components`：棋盘、工具栏、对局面板。
  - `src/game`：规则、棋盘状态、胜负判定、AI。
  - `src/server`：后续实时服务。
  - `src/lib`：通用工具。
- 定义棋盘数据结构：15x15，空位、黑棋、白棋。
- 定义落子事件、胜负结果、对局状态类型。

验收标准：

- 本地能启动开发服务。
- 有一个可渲染的 15x15 空棋盘页面。
- 规则模块有单元测试入口。

预计时间：

- 0.5-1 天。

## 6. 阶段 1：基础可玩版本

目标：

- 先做出完整单机体验，证明棋盘、规则、交互和移动端手感可用。

功能范围：

- 本地双人对弈。
- 人机对战初版。
- 胜负判定。
- 重新开始。
- 悔棋。
- 最后一步高亮。
- 当前回合提示。
- 移动端棋盘自适应。

规则范围：

- 默认普通五子棋：黑先，横竖斜任意五连或以上获胜。
- 先不启用禁手。
- 规则模块预留扩展：Renju、Swap2、长连限制。

AI 初版：

- Easy：随机合法落子，但优先补杀和防守对方四连。
- Normal：基于棋型评分，识别活二、活三、冲四、活四。
- Hard 暂缓到阶段 2 或 3。

技术任务：

- 实现 `createBoard`、`placeStone`、`checkWin`、`getLegalMoves`。
- 胜负检测从最后一步向四个方向扫描，避免每步全盘扫描。
- 实现棋盘点击/触摸坐标转换。
- 实现悔棋和重开状态恢复。
- 加入基础 AI 评分函数。
- 编写规则单元测试。

验收标准：

- 可以完整下完一盘。
- 横、竖、两条斜线均能正确判胜。
- 已有棋子不能重复落子。
- 移动端点击准确，不出现明显误触。
- AI 不会落到已有棋子上。

预计时间：

- 2-4 天。

## 7. 阶段 2：好友房在线对战

目标：

- 支持用户创建房间并分享链接，完成真正的在线对弈 MVP。

功能范围：

- 创建房间。
- 生成房间码和邀请链接。
- 加入房间。
- 双方准备。
- 实时落子同步。
- 认输。
- 对局结束展示胜负。
- 基础断线提示。
- 房主重新开始。

实时事件设计：

- `room:create`
- `room:join`
- `room:leave`
- `player:ready`
- `game:start`
- `game:move`
- `game:resign`
- `game:restart`
- `game:state`
- `connection:lost`
- `connection:recovered`

服务端原则：

- 服务端必须是权威状态，不信任客户端传来的棋盘结果。
- 客户端只提交落子坐标。
- 服务端校验房间、回合、坐标、胜负后广播结果。
- 每个房间最多两个玩家，后续再加观战。

技术任务：

- 建立 Socket.IO 服务。
- 实现 room join/leave。
- 实现服务端房间状态机。
- 实现房间码生成和冲突检查。
- 实现断线后短时间保留座位。
- 前端接入实时事件。
- 加入基础端到端测试或手动测试脚本。

验收标准：

- 两个浏览器窗口可以进入同一房间。
- 任意一方落子，另一方实时看到。
- 非当前回合无法落子。
- 一方断线后另一方看到提示。
- 刷新页面后能尽量恢复当前对局。

预计时间：

- 4-7 天。

## 8. 阶段 3：随机匹配与账号/排行榜

目标：

- 增加留存和复玩动机，让网站从“工具”变成“游戏平台雏形”。

功能范围：

- 游客昵称。
- 可选账号登录。
- 随机匹配。
- 匹配等待页。
- 对局历史。
- 积分/ELO。
- 排行榜。
- 每日榜单。
- 逃跑/中途离开处理。

账号策略：

- 第一版可支持游客，账号作为增强功能。
- 登录方式优先考虑邮箱 magic link 或 OAuth。
- 游客也能玩，但不进入正式排行榜，或只进入临时榜。

排行榜策略：

- 总积分榜。
- 每日胜场榜。
- 连胜榜。
- 可后续按规则模式区分：普通五子棋、禁手、Swap2。

技术任务：

- 接入 PostgreSQL 和 Prisma。
- 设计表：
  - `User`
  - `GuestSession`
  - `Game`
  - `Move`
  - `Rating`
  - `LeaderboardSnapshot`
- Redis 实现匹配队列。
- 实现 ELO 计算。
- 实现异常对局过滤：过短对局、同设备刷分、高频重复匹配。

验收标准：

- 用户能随机匹配到对手。
- 对局结束后积分变化正确。
- 排行榜能展示真实数据。
- 未登录用户也能顺畅体验。

预计时间：

- 5-10 天。

## 9. 阶段 4：广告变现与合规

目标：

- 在不损害体验、不违反广告政策的前提下开始变现。

广告位原则：

- 棋盘周围留出足够空白，广告不能贴近高频点击区域。
- 移动端不在棋盘上方、下方紧贴放高诱导广告。
- 不使用“点这里支持我们”等诱导点击文字。
- 不让广告看起来像游戏按钮、菜单或奖励按钮。
- 插屏只在自然间隙出现：对局结束、重新匹配前、等待页。

初期广告位：

- 桌面端右侧信息栏下方展示广告。
- 桌面端对局结束弹窗外的页面区域展示广告。
- 移动端只在对局结束页、排行榜页、规则页放广告。
- 内容页使用文章中部/底部广告。

合规页面：

- Privacy Policy。
- Terms of Service。
- Cookie/Advertising Notice。
- Contact。
- About。

技术任务：

- 添加广告组件抽象，支持开发环境占位。
- 添加 `ads.txt`。
- 添加隐私政策页面。
- 接入 Google Search Console。
- 接入 Google Analytics 或 Plausible。
- 建立广告展示开关，方便审核前后控制。
- 做移动端广告误触检查。

验收标准：

- 广告不遮挡棋盘和按钮。
- 隐私政策包含 Google 广告 cookie、个性化广告、退出选项等必要说明。
- 页面内容量不低于广告量。
- 开发和测试环境不加载真实广告点击。

预计时间：

- 2-4 天，后续长期优化。

- 让网站具备可持续获取搜索流量的能力，而不是只靠外部推广。

内容页面：

- 五子棋规则。
- Gomoku rules。
- Five in a row online。
- 五子棋技巧。
- 常见棋型：活三、冲四、活四、双三、双四。
- Renju 禁手规则。
- Swap2 规则说明。
- 五子棋和围棋的区别。
- 人机难度说明。
- 在线五子棋 FAQ。

SEO 技术任务：

- 静态生成内容页。
- 添加 sitemap。
- 添加 robots.txt。
- 添加 Open Graph 和 Twitter Card。
- 添加结构化数据。
- 每个内容页链接回游戏页面。
- 游戏页面链接到规则和教程页面。

验收标准：

- 搜索引擎可以抓取主要内容页。
- 内容页不是空壳，有足够原创说明。
- 游戏页首屏可玩，且有可索引规则摘要。

预计时间：

- 3-7 天，之后持续追加。

目标：

- 在核心体验稳定后增加差异化能力。

可选功能：

- Hard AI：minimax + alpha-beta pruning + 棋型评分。
- AI Web Worker，避免阻塞 UI。
- Swap2 模式。
- Renju 禁手模式。
- 观战。
- 对局复盘。
- 分享棋谱。
- 锦标赛。
- 好友系统。
- 会员去广告。
- 主题皮肤。
- 每日残局挑战。

Hard AI 技术方向：

- 候选点只搜索已有棋子周围 2 格内空位。
- 优先检测必胜、必防。
- 棋型评分：五连、活四、冲四、活三、眠三、活二。
- alpha-beta 剪枝。
- 深度按难度调整。
- 后续可加入开局库。

商业化增强：

- 会员去广告。
- 主题皮肤。
- 自定义棋盘。
- 排名徽章。
- 激励广告换复盘次数或皮肤试用，避免影响公平对局。

验收标准：

- 高级功能不能破坏核心对局稳定性。
- 会员/皮肤不影响胜负公平。
- 广告或付费入口不干扰棋盘操作。

预计时间：

- 按功能拆分，每项 2-10 天不等。

## 12. 风险与对策

实时同步风险：

- 风险：客户端状态不一致、重复落子、断线导致胜负争议。
- 对策：服务端权威状态，客户端只提交意图，所有结果由服务端广播。

广告合规风险：

- 风险：广告靠近棋盘导致误触，或页面广告多于内容。
- 对策：广告组件统一控制间距，移动端少放广告，先用内容页提高站点价值。

低价值内容风险：

- 风险：纯游戏页可能不容易通过广告审核。
- 对策：上线前补足规则、技巧、FAQ、隐私政策、关于页面。

作弊和刷榜风险：

- 风险：用户开多窗口刷分、脚本刷局。
- 对策：限制同 IP/设备重复匹配，过滤过短对局，排行榜只统计有效局。

AI 体验风险：

- 风险：AI 太弱留不住人，太强又挫败新手。
- 对策：多难度，默认 Normal，保留 Easy 给新手。

移动端误触风险：

- 风险：15x15 棋盘在小屏幕点击不准。
- 对策：棋盘最大化、落子前高亮交点、必要时支持放大或确认落子。

## 13. 推荐里程碑

M1：本地可玩

- 完成本地双人、人机 Easy/Normal、胜负判定、移动端棋盘。

M2：好友房上线

- 完成创建房间、邀请链接、实时落子、断线提示。

M3：公开测试

- 部署测试站，邀请用户试玩，记录移动端误触和断线问题。

M4：广告审核准备

- 完成隐私政策、服务条款、内容页、广告占位、Search Console。

M5：正式上线

- 开启基础广告，持续观察留存、局数、广告展示、无效流量风险。

M6：增长增强

- 上线随机匹配、排行榜、SEO 内容、多语言和高级 AI。

## 15. 成功指标

技术指标：

- 页面首屏加载快。
- 棋盘点击准确。
- 对局状态不会错乱。
- 手机浏览器体验稳定。

产品指标：

- 用户能在 5 秒内开始第一盘。
- 好友邀请链路不超过 2 步。
- 用户愿意连续玩 2-3 局。
- 移动端退出率可控。

商业指标：

- 广告通过审核。
- 无明显无效流量告警。
- 每局广告曝光不影响继续对局。
- SEO 页面逐步带来自然访问。
