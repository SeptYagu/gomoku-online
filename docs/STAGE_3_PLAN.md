# Stage 3 Plan: Lobby, Spectators, Chat, Matchmaking, Rankings

更新日期：2026-06-25

阶段 3 在 M3 公开测试收口后启动。目标不是先做重账号系统，而是先把好友房从“只能靠房间码私下约人”升级成一个轻量游戏平台雏形：可分享、可发现房间、可观战、可聊天、可随机匹配，并为后续排行榜和账号沉淀数据。

## 参考方向

PlayOK 当前 gomoku 页面列出的成熟能力包括 live opponents、game rooms、rankings、stats、profiles、contacts、private messaging、game records 和 mobile support。阶段 3 要覆盖这些平台能力里的核心路径：用户状态、Profile、Ranking、Game records、房间列表、观战和聊天。关于 PlayOK 社交结构的研究资料还描述了大厅房间里的活动桌列表、房间在线用户列表、公共文本聊天，以及单桌内的成员列表、玩家座位和桌内聊天。

本项目只参考产品结构和交互边界，不复制 PlayOK 的 UI、代码、文案或数据。

## 阶段 3 新增范围

### 1. 真实分享链接

当前分享按钮不能再只是一个形式按钮。创建或加入房间后，浏览器地址必须进入可分享状态：

- 创建房间成功后，将当前地址更新为包含房间号的 URL，例如 `/en?room=ABC123`。
- 通过房间号加入成功后，也同步更新当前地址。
- Share / Copy invite 按钮复制当前浏览器地址，而不是另行拼一个可能过期的假链接。
- 离开房间后清理 URL 上的 room 参数，避免下次刷新误入旧房。
- 复制失败时给出可手动复制的文本状态。

验收：

- A 创建房间后地址栏立即出现 room code。
- A 复制当前地址发给 B，B 打开后能直接加入或观战同一房间。
- 刷新页面后仍能用 localStorage session 恢复原座位。

### 2. 观战席

房间不应只支持两名成员。第三个及之后进入同一房间的人进入观战席：

- `players` 仍只保留黑白两个对局座位。
- 新增 `spectators` 或等价快照字段，保存观战者昵称、连接状态和加入时间。
- 观战者不能 ready、落子、认输、请求悔棋或响应悔棋。
- 观战者可以看到棋盘、玩家列表、胜负、重开状态和房间聊天。
- 若后续支持空位入座，必须走明确的 `sit` 动作，不能自动抢玩家座位。

验收：

- 第三人打开分享链接后不会挤掉玩家。
- 玩家操作仍只由黑白两席决定。
- 玩家和观战者都能看到观战人数。

### 3. 房间列表 / 大厅

需要一个公开大厅，让玩家看到当前谁在开房间：

- 房间列表显示公开等待房和可观战对局。
- 每行至少显示房主昵称、房间号、状态、玩家数、观战人数、创建时间或最近活动时间。
- waiting 房显示 Join；playing 房显示 Watch。
- 私密房后续可以只显示 hasPassword，不下发密码或 invite token。
- 大厅通过 Socket.IO 增量更新，客户端发现版本缺口时重新拉取列表。

验收：

- 新建公开房间后，大厅其他客户端能看到。
- 房间开始、结束、空房清理后，大厅状态同步更新。
- 点击 Join/Watch 能进入对应房间。

### 4. 聊天频道

阶段 3 需要两个文本频道：

- 公共聊天频道：大厅范围，所有在线大厅用户可见。
- 房间聊天频道：房间范围，玩家和观战者可见。

第一版约束：

- 游客也能聊天，但使用当前随机昵称。
- 单条消息限制长度，建议 200 字符以内。
- 服务端做 trim、空消息拒绝、频率限制和基础 HTML 转义/纯文本传输。
- 公共聊天和房间聊天都保留短期内存历史，例如最近 50 条；持久化可留到账号/审计阶段。
- UI 上聊天区域不能挤压棋盘主操作，移动端优先折叠。

验收：

- 两个大厅客户端能互相看到公共聊天。
- 同一房间内玩家和观战者能看到房间聊天。
- 不同房间的房间聊天互相隔离。
- 过长、过快、空消息被服务端拒绝。

### 5. 随机匹配

在房间列表稳定后做随机匹配：

- Find Match 优先加入可用公开 waiting 房。
- 没有可用房时创建公开 waiting 房。
- 并发匹配不能超员。
- 私密房和满员房不参与随机匹配。

### 6. 排行榜与账号

排行榜和账号仍属于阶段 3，但排在大厅、观战、聊天、匹配之后：

- 第一版继续允许游客完整游玩。
- 正式排行榜只统计注册用户或可信 guest session，具体范围要在 UI 上明确。
- ELO、每日胜场、连胜榜需要服务端权威结算。
- 过短对局、同设备刷分、高频重复匹配要过滤。
- 用户状态需要展示在线、游戏中、观战中、离线等基础状态。
- 注册玩家拥有 Profile，可展示昵称、头像、Rating、统计、最近对局和 Game records。
- Ranking 页面展示总榜、每日榜、连胜榜，并支持从榜单进入 Profile。

### 7. 棋谱提交、去重和回看

每一局在线玩家对局结束后都要提交棋谱到服务器，注册玩家和游客都提交：

- 对局结束时，胜负双方客户端各自向服务器提交一份 game record。
- 服务器用 `roomId/gameId + moveSeq + final board/result` 或服务端签名的 `gameRecordId` 做幂等键。
- 如果只收到一方提交，先保存为 `partial`。
- 如果双方都提交，执行去重和一致性校验；一致则合并为一份 `verified` game record。
- 如果双方提交不一致，以服务端权威房间快照为准，并把冲突标记为 `conflicted` 供后续审计。
- 注册玩家的棋谱进入个人 Game records，可从 Profile 回看。
- 游客棋谱也提交并保存为匿名/guest game record，进入服务器棋谱池，用于后续离线分析、开局库生成和总体统计；默认不提供个人云端回看，不挂到公开 Profile，也不进入正式注册玩家 Ranking。
- 棋谱保存格式优先使用 SGF 或可无损转换为 SGF 的结构化 JSON；后续开局库生成以服务器保存棋谱为输入。
- 隐私上不保存不必要的 IP 明文；如需反作弊，只保存最小化 hash/设备指纹并在隐私说明中披露。

验收：

- 注册玩家对局结束后，两端都提交棋谱，服务器最终只保留一份 verified 记录。
- 游客对局结束后也提交棋谱，服务器保存匿名记录，用于后续离线分析和开局库生成。
- 重复提交同一局不会产生重复 game record。
- 注册玩家可在自己的 Profile / Game records 中回看棋谱。
- 开局库工具能从保存的 game records 中读取候选棋谱。

## 建议小步顺序

1. 分享链接修正：创建/加入后更新 URL，分享按钮复制当前 URL。
2. 房间成员模型扩展：支持 spectators，第三人进入观战席。
3. 房间列表 API 和 lobby socket channel。
4. 房间列表 UI：Join / Watch。
5. 房间聊天频道。
6. 公共聊天频道。
7. 随机匹配。
8. 在线棋谱提交、去重和匿名/注册记录保存。
9. 注册玩家 Profile 和 Game records 回看。
10. 排行榜。
11. 可选账号登录。

执行进度见 `docs/STAGE_3_PROGRESS.md`。

每个小步都必须：

- 更新相关文档。
- 通过 `npm test`、`npm run lint`、`npm run build`。
- 提交并推送 GitHub。
- 推送后等待 90 秒，再运行线上版本检查。
