# 标准开发流程

更新日期：2026-06-23

本项目以后默认采用“主控 + 实现子代理 + 独立验证子代理”的工作模式。主控负责拆解、派发、整合和最终判断；实现子代理负责具体代码或文档改动；验证子代理负责独立测试、真实浏览器检查和风险回归。

## 1. 核心原则

- 主控不包办实现。主控负责清楚地定义目标、边界、验收标准和任务拆分。
- 一个子代理只负责一个清晰模块或一个不重叠的文件范围。
- 实现和验证必须分离。写代码的人不做唯一验收人。
- 研究、实现、验证、交接都要留下可追溯文档。
- 无许可证项目只参考流程和设计，不复制源码。
- 每完成一版都提交并推送到 `origin/main`。

## 2. 标准角色

### 主控

职责：

- 理解用户目标。
- 拆分模块和任务。
- 判断是否需要研究子代理。
- 派发实现子代理。
- 派发独立验证子代理。
- 合并子代理结果。
- 处理冲突和最终决策。
- 更新计划、标准流程、交接文档。
- 最终提交和推送。

主控应该避免：

- 在可拆分任务中直接自己写大量代码。
- 让同一个代理既实现又验收。
- 把多个模块混给一个子代理。
- 在未定义验收标准前开始实现。

### 研究子代理

职责：

- 深读参考项目或本项目某个模块。
- 输出关键文件、行号、状态流、数据结构、风险和可采用方案。
- 不修改文件，除非主控明确授权写入范围。

输出必须包含：

- 相关文件和关键行号。
- 参考实现细节。
- 风险和不能照搬的地方。
- 本项目改造方案。
- 测试清单。
- 许可证边界。

### 实现子代理

职责：

- 按主控指定范围修改代码或文档。
- 只改自己负责的文件或模块。
- 遇到用户或其他代理的改动时，不回滚，先适配。
- 在 final 中列出修改的文件和验证命令。

派发时必须说明：

- 目标。
- 写入范围。
- 禁止修改的范围。
- 依赖文档。
- 验收标准。
- 需要运行的最小验证。

### 验证子代理

职责：

- 不参与实现。
- 独立检查实现结果。
- 运行测试、lint、build。
- 对前端改动进行真实浏览器或截图级检查。
- 报告问题、风险和未覆盖项。

验证子代理必须尽量覆盖：

- 自动化测试。
- 类型检查或构建。
- lint。
- 浏览器可用性。
- 移动端布局。
- 多语言和 RTL。
- 浅色/黑暗模式。
- 广告误触风险。

## 3. 任务生命周期

### Step 1：定义任务

主控先写清楚：

- 用户目标。
- 涉及模块。
- 不做什么。
- 验收标准。
- 风险点。

如果目标不清楚，先追问；如果能合理假设，则记录假设后继续。

### Step 2：研究

满足以下任一条件时先研究：

- 要参考外部项目。
- 涉及实时同步、AI、评分、广告、支付、账号、安全、SEO。
- 许可证不清晰。
- 代码影响范围较大。

研究流程遵循 `docs/STANDARD_RESEARCH_WORKFLOW.md`。

### Step 3：拆分实现

按不重叠写入范围拆分。

示例：

- i18n 基础设施。
- 主题 token 和切换。
- 棋盘规则 API。
- AI Easy/Normal。
- Socket.IO 房间服务。
- 验证测试。

每个实现子代理只拥有自己的文件范围。

### Step 4：实现

实现子代理修改代码或文档。

主控在实现期间做：

- 回收已完成子代理。
- 检查是否有冲突。
- 整合文档和代码。
- 保持任务清单更新。

### Step 5：独立验证

实现完成后，必须派验证子代理。

验证子代理默认运行：

```bash
npm test
npm run lint
npm run build
npm audit --omit=dev
```

前端体验相关改动还必须检查：

- 桌面浏览器。
- 移动端视口。
- 点击/触摸目标。
- 浅色/黑暗模式。
- 六种语言。
- 阿拉伯语 RTL。
- 文字是否溢出。
- 广告区域是否远离棋盘和按钮。

### Step 6：主控收口

主控负责：

- 修复验证发现的问题，或重新派实现代理。
- 更新 `docs/HANDOFF.md`。
- 更新相关计划/模块文档。
- 运行必要的最终验证。
- 提交并推送。

## 4. Definition of Ready

任务开始实现前，至少满足：

- 目标明确。
- 涉及文件或模块明确。
- 验收标准明确。
- 许可证风险已判断。
- 如果需要参考项目，模块研究文档已存在或正在由研究子代理完成。
- 如果是前端功能，已明确桌面/移动/主题/语言验收范围。

## 5. Definition of Done

一项任务完成必须满足：

- 功能或文档已实现。
- 实现范围没有越界。
- 自动化验证通过，或明确记录未运行原因。
- 前端改动完成真实浏览器检查，或明确记录工具阻塞原因。
- 相关文档已更新。
- `docs/HANDOFF.md` 已更新。
- 工作区干净。
- 已提交并推送。

## 6. 质量门禁

默认门禁：

```bash
npm test
npm run lint
npm run build
npm audit --omit=dev
```

风险较高时追加：

- 单元测试覆盖核心规则。
- 端到端测试覆盖主要用户路径。
- 浏览器截图检查。
- 移动端视口检查。
- 性能预算检查。
- 安全/隐私检查。

## 7. 前端真实浏览器验收

涉及 UI、布局、主题、语言、棋盘交互时，必须做真实浏览器验收。

至少检查：

- 1440px 桌面。
- 1024px 平板。
- 390px 手机。
- 浅色模式。
- 黑暗模式。
- 英语默认。
- 中文。
- 法语。
- 西班牙语。
- 俄语。
- 阿拉伯语 RTL。

五子棋页面额外检查：

- 棋盘非空白。
- 棋盘比例稳定。
- 点击/触摸准确。
- 棋子不改变格子尺寸。
- 最后一步和胜线清楚。
- 文字不遮挡棋盘。
- 广告位不靠近高频点击区域。

## 8. 文档制度

必须长期维护：

- `WEBSITE_BUILD_PLAN.md`：总体计划。
- `docs/REUSE_EVALUATION.md`：复用与许可证边界。
- `docs/STANDARD_RESEARCH_WORKFLOW.md`：参考项目研究流程。
- `docs/STANDARD_DEVELOPMENT_WORKFLOW.md`：开发协作流程。
- `docs/HANDOFF.md`：当前任务交接。
- `docs/logic/*.md`：模块级研究记录。

重要设计决策建议新增 ADR：

```text
docs/adr/0001-title.md
```

ADR 应包含：

- 背景。
- 决策。
- 备选方案。
- 影响。
- 回滚条件。

## 9. 安全与合规

默认规则：

- 不提交私钥、APK、研究仓库、日志。
- 不把广告放在棋盘或按钮附近。
- 不在客户端信任胜负、评分、房间状态。
- 不向客户端下发密码明文或敏感用户字段。
- 不复制无许可证项目源码。
- MIT 代码迁移必须补第三方声明。

## 10. 发布与回滚

每个版本：

- 小步提交。
- 提交信息清楚。
- 推送到 GitHub。
- 记录验证结果。

上线前：

- 确认环境变量。
- 确认广告开关。
- 确认隐私政策。
- 确认 sitemap 和 `hreflang`。

回滚策略：

- 优先用 Git 历史回滚具体提交。
- 不使用破坏性命令清空用户改动。
- 回滚前记录原因和影响范围。

## 11. 当前项目特别规则

- 默认语言是 English。
- 必须支持联合国六种官方语言：`en`、`zh`、`fr`、`es`、`ru`、`ar`。
- 阿拉伯语必须支持 RTL，但棋盘不镜像。
- 必须支持浅色/黑暗模式。
- 在线对战必须服务端权威。
- 广告不能影响棋盘操作。
- 每个模块开发前先读对应 `docs/logic/*.md`。
