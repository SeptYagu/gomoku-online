# 标准开发流程

更新日期：2026-06-25

本项目默认采用“主控 + 研究子代理 + 实现子代理 + 独立验证子代理”的工作模式。主控负责目标判断、任务拆分、代理派发、结果整合、最终验收、提交和推送；子代理负责被分配的清晰任务。除非主控明确授权，子代理不提交、不推送、不改超出范围的文件。

## 1. 硬性原则

- 主控默认不直接承担大块功能实现，功能代码由实现子代理按模块完成。
- 主控可以做任务拆分、冲突仲裁、少量整合修补、文档收口、最终验证和提交推送。
- 一个实现子代理只拥有一个清晰模块或一个不重叠写入范围。
- 实现和验证必须分离，写代码的人不能作为唯一验收人。
- 每个阶段都必须留下可追溯产物：研究记录、实现摘要、验证报告、交接更新。
- 所有子代理报告必须写成独立文档，不能只留在聊天记录里。
- 每次使用子代理后，必须把该子代理的报告落到 `docs/subagents/` 或主控明确指定的同类目录。
- 子代理报告文档的文件名和文档开头必须使用时间编号，方便排序、溯源和交接。
- 无许可证项目只参考流程、交互、状态机和风险，不复制源码、样式、素材或数据。
- 每完成一版，由主控统一检查、提交并推送到 `origin/main`。

## 2. 标准角色

### 主控

职责：

- 理解用户目标并记录关键假设。
- 定义任务边界、验收标准和不做事项。
- 判断是否需要研究子代理。
- 拆分不重叠的实现任务。
- 派发研究、实现和验证子代理。
- 整合子代理结果，处理冲突和最终取舍。
- 更新计划、模块文档、流程规范和 `docs/HANDOFF.md`。
- 运行最终门禁，提交并推送。

主控禁止：

- 在可拆分功能中长期替代实现子代理写代码。
- 让同一个子代理同时负责实现和最终验收。
- 把多个高耦合模块混给一个子代理。
- 在未定义验收标准前开始实现。
- 在验证子代理发现 blocker 或 major 问题后，亲自替代实现子代理修复。
- 未检查用户或其他代理改动就回滚文件。

### 研究子代理

职责：

- 深读参考项目或本项目某个模块。
- 输出关键文件、行号、状态流、数据结构、风险和可采用方案。
- 记录许可证边界和不能复制的内容。
- 不修改文件，除非主控明确给出写入范围。

输出必须包含：

- 相关文件和关键行号。
- 参考实现细节。
- 风险和不能照搬的地方。
- 本项目改造方案。
- 测试清单。
- 许可证边界。

### 实现子代理

职责：

- 按主控指定范围修改代码或文档。
- 只改自己负责的文件或模块。
- 遇到用户或其他代理的改动时，不回滚，先适配。
- 在最终回复中列出修改文件、验证命令、未覆盖风险。

实现子代理默认不做：

- 不提交。
- 不推送。
- 不改共享文件，除非派发任务明确授权。
- 不改其他代理拥有的范围。

### 验证子代理

职责：

- 不参与实现。
- 独立检查实现结果。
- 运行测试、lint、build、必要的安全检查。
- 对前端改动做真实浏览器或截图级检查。
- 报告失败证据、风险和未覆盖项。

验证子代理默认不修代码。如果需要修复，主控必须重新派实现子代理，并把失败报告作为返工输入；修复后必须再派独立验证子代理复测。

## 3. 任务生命周期

### Step 1：任务定义

主控先写清楚：

- 用户目标。
- 涉及模块。
- 不做什么。
- 验收标准。
- 风险点。
- 是否需要研究。
- 是否需要真实浏览器验证。

如果目标不清楚，先追问；如果能合理假设，则记录假设后继续。

### Step 2：研究

满足以下任一条件时先研究：

- 要参考外部项目。
- 涉及实时同步、AI、评分、广告、支付、账号、安全、SEO。
- 许可证不清晰。
- 代码影响范围较大。

研究流程遵循 `docs/STANDARD_RESEARCH_WORKFLOW.md`。

### Step 3：拆分实现

主控按不重叠写入范围拆分任务。示例：

- i18n 基础设施。
- 主题 token 和切换。
- 棋盘规则 API。
- AI Easy/Normal。
- Socket.IO 房间服务。
- 验证测试。

拆分时必须标出共享文件和依赖顺序。

### Step 4：实现

实现子代理修改代码或文档。主控在实现期间负责：

- 回收已完成子代理结果。
- 检查是否有文件范围冲突。
- 整合文档和代码。
- 保持任务清单和交接文档更新。
- 确认每个实现子代理都提交独立报告文档。

### Step 5：独立验证

实现完成后，必须派独立验证子代理。验证子代理默认运行：

```bash
npm test
npm run lint
npm run build
npm audit --omit=dev
```

前端体验相关改动还必须检查：

- 桌面浏览器。
- 移动端视口。
- 点击/触摸目标。
- 浅色/黑暗模式。
- 六种语言。
- 阿拉伯语 RTL。
- 文字是否溢出。
- 广告区域是否远离棋盘和按钮。

### Step 6：返工或收口

如果验证失败：

- 验证子代理必须提交失败报告。
- 失败报告必须落为独立文档，不能只留在聊天记录。
- 主控判断 blocker、major、minor。
- blocker 和 major 必须返工后复测。
- 主控不得亲自补 blocker 或 major；必须按失败报告重新派实现子代理修复。
- 返工实现子代理必须在报告中引用失败报告文档路径。
- 修复后必须派独立验证子代理复测，且复测必须覆盖原失败清单。
- minor 如被主控降级延期，必须在交接文档中记录原因、影响和后续 owner。

如果验证通过：

- 主控更新 `docs/HANDOFF.md`。
- 主控更新相关计划、模块文档或 ADR。
- 主控运行最终门禁。
- 主控提交并推送。

## 4. 子代理派发模板

每次派发任务应使用以下结构：

```text
任务 ID：
角色：研究 / 实现 / 验证
目标：
背景：
允许修改范围：
禁止修改范围：
共享文件和依赖：
必须阅读文档：
验收标准：
必须运行命令：
输出格式：
报告文档路径：
未完成时如何汇报：
```

实现类任务必须明确文件 owner。验证类任务必须明确要复测的行为和证据格式。所有派发任务必须明确报告文档路径；如果主控允许子代理自行命名，也必须要求其遵循 `docs/subagents/` 的命名规范。

## 5. 子代理报告目录和命名规范

默认目录：

```text
docs/subagents/
```

每次使用研究、实现或验证子代理后，必须新增或更新一份独立报告文档。聊天记录只能作为临时沟通，不视为可追溯产物。

文件名建议格式：

```text
YYYYMMDD-HHMMSS-任务ID-代理角色-模块-agentId或昵称-状态.md
```

允许在任务已经有唯一日期编号时使用短格式：

```text
YYYYMMDD-任务ID-代理角色-模块-agentId或昵称.md
```

文档开头必须包含：

```text
时间编号：
任务 ID：
代理角色：
agent id：
昵称：
状态：完成 / 部分完成 / 失败 / 阻塞
写入范围：
```

报告正文至少包含：

- 任务目标。
- 实际完成。
- 修改文件或阅读文件。
- 验证命令和结果。
- 失败证据或未覆盖项。
- 交给主控或下一位子代理的建议。

主控在更新 `docs/HANDOFF.md` 时，必须记录本轮子代理报告文档路径。

## 6. 并行协作与共享文件规则

共享文件包括但不限于：

- `package.json`
- `package-lock.json`
- `src/app/**`
- `src/app/globals.css`
- i18n 字典和路由文件。
- 主题 token 文件。
- 全局类型文件。
- Socket 事件类型。
- Prisma schema 或数据库迁移。
- `README.md`
- `WEBSITE_BUILD_PLAN.md`
- `docs/HANDOFF.md`

规则：

- 共享文件默认由主控串行修改，或只授权给一个实现子代理。
- 如果两个任务都需要同一共享文件，主控必须安排先后顺序。
- 子代理开始前应检查 `git status --short`。
- 子代理完成后必须报告实际修改文件，并把结果写入独立报告文档。
- 主控合并前必须看 `git status --short` 和 `git diff --stat`。
- 冲突时优先保留用户改动，其次保留已验证实现，最后由主控重新派返工任务。

## 7. 验证失败返工模板

验证失败必须按以下格式记录：

```text
问题：
严重级别：blocker / major / minor / note
复现步骤：
期望结果：
实际结果：
证据：命令输出 / 截图路径 / 浏览器视口 / 日志
影响范围：
建议返工范围：
必须复测清单：
失败报告文档路径：
```

严重级别定义：

- blocker：核心功能不可用、构建失败、测试失败、安全风险、数据破坏、广告严重误触。
- major：主要路径有缺陷、移动端或多语言明显不可用、规则错误、状态同步风险。
- minor：可绕过的小问题、文案或样式瑕疵，不阻塞阶段交付但必须记录。
- note：建议或后续优化。

验证失败后的硬规则：

- blocker 和 major 必须由实现子代理返工。
- 主控只做调度、整合、少量必要衔接和最终提交。
- 返工完成后必须由独立验证子代理复测。
- 复测报告必须引用原失败报告文档路径和返工报告文档路径。

## 8. Definition of Ready

任务开始实现前，至少满足：

- 目标明确。
- 涉及文件或模块明确。
- 验收标准明确。
- 许可证风险已判断。
- 如果需要参考项目，模块研究文档已存在或正在由研究子代理完成。
- 如果是前端功能，已明确桌面、移动、主题、语言验收范围。
- 如果涉及共享文件，已指定 owner 和执行顺序。

## 9. Definition of Done

一项任务完成必须满足：

- 功能或文档已实现。
- 实现范围没有越界。
- 自动化验证通过，或明确记录未运行原因。
- 前端改动完成真实浏览器检查，或明确记录工具阻塞原因。
- 验证失败项已返工并复测，或被主控明确降级记录。
- 相关文档已更新。
- `docs/HANDOFF.md` 已更新。
- 本轮所有子代理报告文档路径已写入 `docs/HANDOFF.md`。
- 主控确认工作区干净。
- 主控已提交并推送。

## 10. 质量门禁

默认门禁：

```bash
npm test
npm run lint
npm run build
npm audit --omit=dev
```

依赖审计规则：

- critical 和 high 必须修复，或记录明确豁免原因。
- moderate 和 low 可以记录后继续，但不能隐瞒。
- 生产依赖风险优先级高于开发依赖风险。

风险较高时追加：

- 单元测试覆盖核心规则。
- 端到端测试覆盖主要用户路径。
- 浏览器截图检查。
- 移动端视口检查。
- 性能预算检查。
- 安全/隐私检查。
- secrets scan 或等价检查。

## 11. 前端真实浏览器验收

涉及 UI、布局、主题、语言、棋盘交互时，必须做真实浏览器验收。

至少检查：

- 1440px 桌面。
- 1024px 平板。
- 390px 手机。
- 浅色模式。
- 黑暗模式。
- 英语默认。
- 中文。
- 法语。
- 西班牙语。
- 俄语。
- 阿拉伯语 RTL。

五子棋页面额外检查：

- 棋盘非空白。
- 棋盘比例稳定。
- 点击/触摸准确。
- 棋子不改变格子尺寸。
- 最后一步和胜线清楚。
- 文字不遮挡棋盘。
- 广告位不靠近高频点击区域。

验证证据应包含：

- 运行命令摘要。
- 浏览器 URL。
- 视口尺寸。
- 语言和主题组合。
- 截图路径或明确说明未能截图的原因。

## 12. 文档更新映射

必须长期维护：

- `WEBSITE_BUILD_PLAN.md`：总体计划、阶段范围、商业化策略。
- `docs/REUSE_EVALUATION.md`：复用与许可证边界。
- `docs/STANDARD_RESEARCH_WORKFLOW.md`：参考项目研究流程。
- `docs/STANDARD_DEVELOPMENT_WORKFLOW.md`：开发协作流程。
- `docs/HANDOFF.md`：当前任务交接。
- `docs/subagents/`：子代理独立报告归档。
- `docs/logic/*.md`：模块级研究记录。

按变更类型更新：

- 规则引擎：更新 `docs/logic/rules-engine-module.md`。
- 在线房间：更新 `docs/logic/realtime-room-module.md`。
- 大厅匹配：更新 `docs/logic/lobby-matchmaking-module.md`。
- AI：更新 `docs/logic/ai-engine-module.md` 和必要的 Worker 文档。
- 多语言/主题：更新 `docs/logic/i18n-theme-module.md`。
- 排行榜：更新 `docs/logic/rating-leaderboard-module.md`。
- 架构级不可逆决策：新增 ADR。

ADR 路径：

```text
docs/adr/0001-title.md
```

ADR 应包含：

- 背景。
- 决策。
- 备选方案。
- 影响。
- 回滚条件。

## 13. 安全与合规门禁

默认检查：

- 不提交私钥、APK、研究仓库、日志、`.env`。
- 不把广告放在棋盘或按钮附近。
- 不在客户端信任胜负、评分、房间状态。
- Socket 事件必须校验身份、房间成员、回合、落子合法性。
- 服务端必须权威结算胜负和评分。
- 用户输入必须校验和限流。
- 不向客户端下发密码明文或敏感用户字段。
- 不复制无许可证项目源码。
- MIT 代码迁移必须补第三方声明。
- 涉及账号、广告、分析、cookie 时必须补隐私说明。

## 14. 提交与推送门禁

只有主控执行提交和推送，除非用户明确授权其他代理。

提交前必须检查：

```bash
git status --short --branch
git diff --check
```

代码改动还必须运行与风险相称的项目门禁。文档-only 改动至少运行 `git diff --check`。

推送规则：

- 禁止 `git push --force`，除非用户明确要求并理解后果。
- 推送前确认没有用户未授权改动被混入提交。
- 如果远端有新提交，优先使用 `git pull --ff-only`。
- 如果发生冲突，停止并由主控仲裁，不强行覆盖。
- CI 或部署失败时，不得把任务视为完成。

## 15. 当前项目特别规则

- 默认语言是 English。
- 必须支持联合国六种官方语言：`en`、`zh`、`fr`、`es`、`ru`、`ar`。
- 阿拉伯语必须支持 RTL，但棋盘不镜像。
- 必须支持浅色/黑暗模式。
- 在线对战必须服务端权威。
- 广告不能影响棋盘操作。
- 每个模块开发前先读对应 `docs/logic/*.md`。
