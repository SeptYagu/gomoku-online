# 标准参考项目研究流程

更新日期：2026-06-23

本项目以后研究、复用或参考外部项目时，统一使用这套流程。目标是把“看过参考项目”升级成可审计、可追溯、可执行的模块级研究结果。

本流程服从 `docs/STANDARD_DEVELOPMENT_WORKFLOW.md`：主控负责派发和整合，研究子代理负责模块深读，独立验证子代理负责后续验证。研究子代理默认不提交、不推送。

## 1. 触发条件

出现以下情况时必须启动本流程：

- 需要参考外部开源项目实现某个模块。
- 用户明确要求学习、研究、复用或参考某个项目。
- 某个模块复杂度较高，需要先拆清状态机、算法、数据模型或边界条件。
- 参考项目许可证不明确，需要避免误复制源码。
- 涉及实时同步、AI、评分、广告、账号、安全、SEO 或商业化。

## 2. 模块拆分

研究必须按模块拆分，不能把所有项目混成一篇粗略总结。

当前模块划分：

- 实时好友房：`docs/logic/realtime-room-module.md`
- 大厅与匹配：`docs/logic/lobby-matchmaking-module.md`
- 评分与排行榜：`docs/logic/rating-leaderboard-module.md`
- 规则引擎：`docs/logic/rules-engine-module.md`
- AI 引擎：`docs/logic/ai-engine-module.md`
- AI Worker 与设置持久化：`docs/logic/ai-worker-persistence-module.md`
- 多语言与主题：`docs/logic/i18n-theme-module.md`

如果后续新增模块，必须新建单独文件，不把内容塞进已有无关文档。

## 3. 研究来源锁定

每个参考来源必须记录固定版本，避免后续无法审计。

来源模板：

```text
项目名：
URL：
访问日期：
commit / tag / release：
LICENSE 路径：
许可证结论：可复用 / 只能参考 / 不可使用
允许复用范围：
禁止复用范围：
需要署名或声明的位置：
证据文件和关键行号：
```

如果无法确认 commit 或 LICENSE，必须标记为“只能参考，不复制源码”。

## 4. 子代理分工

当用户要求使用子代理，或模块数量较多时，使用一个子代理研究一个模块。

原则：

- 一个子代理只负责一个模块。
- 子代理只阅读该模块相关项目和文件。
- 子代理不修改代码和文档，除非主控明确分配写入范围。
- 子代理输出必须包含源码路径和关键行号。
- 子代理不能复制无许可证项目源码，只能总结流程、状态机、数据模型和风险。
- MIT 或其他许可允许复用的项目，也必须记录署名和许可证要求。

## 5. 研究子代理派发模板

```text
任务 ID：
研究模块：
参考项目：
必须查看的目录或文件：
禁止事项：
需要回答的问题：
输出格式：
许可证检查要求：
是否允许写文件：默认否
```

输出必须包含：

- 来源锁定信息。
- 相关文件清单，带路径和关键行号。
- 模块流程或算法细节。
- 状态变化、事件、数据结构或函数职责。
- 参考实现的优点。
- 参考实现的风险或不能照搬的地方。
- 本项目应采用的改造方案。
- 可执行实现任务清单。
- 测试清单。
- 许可证和不复制边界。

## 6. 主控整合

主控收到子代理结果后，必须做整合，而不是原样粘贴。

整合要求：

- 合并重复结论。
- 把源码观察转成适合本项目的 TypeScript、Next.js、Socket.IO、Prisma、Redis 方案。
- 明确客户端和服务端责任边界。
- 明确哪些逻辑可以迁移、哪些只能参考。
- 保留关键路径和行号，方便日后回查。
- 把研究结果转成优先级明确的开发任务。
- 更新对应 `docs/logic/*.md` 文件。

## 7. 文档质量标准

模块文档不能只写“适合参考”这种粗略描述，必须达到以下标准：

- 能说清楚模块的入口函数、事件或页面。
- 能说清楚数据如何流动。
- 能说清楚状态如何变化。
- 能说清楚边界条件和错误处理。
- 能说清楚为什么某些实现不能直接采用。
- 能直接转化为后续开发任务。

每个模块文档至少包含：

- `参考来源`
- `来源锁定`
- `关键文件和证据`
- `参考实现细节`
- `风险与不足`
- `本项目采用方案`
- `实现任务清单`
- `测试清单`
- `许可证边界`

## 8. 许可证规则

无明确 LICENSE：

- 不复制源码。
- 不迁移组件、函数、样式或数据模型代码。
- 只学习产品路径、交互流程、状态机、事件设计和风险。

MIT 或其他允许复用许可证：

- 可以迁移或改写代码。
- 必须保留许可证文本或第三方声明。
- 必须在模块文档里记录项目名、作者、来源和许可证。
- 迁移代码时优先转成项目自己的 TypeScript 类型、测试和目录结构。

本地 APK：

- 只能作为体验参考。
- 不反编译复用代码或素材。
- 不提交 APK 到源码仓库。

## 9. 研究到实现的转换

研究完成后，主控必须把结果整理成实现任务：

```text
任务名称：
优先级：
依赖任务：
允许修改范围：
禁止修改范围：
输入文档：
验收标准：
验证命令：
风险：
```

如果研究发现当前计划不合理，主控应先更新 `WEBSITE_BUILD_PLAN.md` 或新增 ADR，再派实现任务。

## 10. 研究验证

研究文档入库前，主控至少检查：

- 是否每个结论都有来源或明确标记为推断。
- 是否记录 LICENSE 和 commit/tag。
- 是否没有复制无许可证源码。
- 是否列出本项目采用方案，而不是只总结参考项目。
- 是否能直接指导下一步实现。
- 是否更新 `docs/HANDOFF.md`。

必要时可派独立验证子代理复审研究文档。

## 11. 入库和推送

研究流程完成后，主控负责：

- 更新对应 `docs/logic/*.md`。
- 必要时更新 `WEBSITE_BUILD_PLAN.md`、`README.md`、`docs/REUSE_EVALUATION.md`。
- 更新 `docs/HANDOFF.md`，记录本轮研究完成了什么、下一步怎么接。
- 运行 `git diff --check`。
- 如果只改文档，不需要跑完整前端构建。
- 提交并推送到 `origin/main`。

提交信息建议：

```text
docs: deepen reference research for <module>
```

或多个模块一起：

```text
docs: deepen module research workflow
```
