# 竞品交互设计研究

更新日期：2026-07-10

## 目的与边界

本文只研究竞品的产品路径、界面分工、状态切换、信息优先级和用户操作习惯，用于指导本项目后续交互重构。

- 不以配色、阴影或按钮皮肤作为主要评价标准。
- 无明确许可证的产品或仓库只参考思路，不复制代码、样式、素材、文案或数据。
- 每研究完一个产品立即追加一节，并保留来源版本和证据路径。
- 最终结论要能转化为本项目的页面状态、组件边界和验收任务。

## PlayOK Gomoku

### 来源锁定

- 项目：PlayOK Gomoku
- 官方入口：https://www.playok.com/en/gomoku/
- 官方对局截图：https://www.playok.com/gfx/shot/gm.png
- 当前公开客户端资源：https://www.playok.com/j/gm.js?264
- 辅助流程证据：
  - https://stardasun.tistory.com/28
  - https://dday.tistory.com/422
  - https://www.katch.ne.jp/~vert-bx/playOK.htm
- 访问日期：2026-07-10
- 版本标识：公开客户端资源查询版本 `gm.js?264`；站点没有公开 commit/tag。
- LICENSE：未发现允许复制其前端代码、样式或素材的许可证。
- 许可证结论：只能参考产品结构、交互流程和状态边界，不复制实现。

### 一句话结论

PlayOK 的价值不在视觉精致，而在于它把“大厅”和“牌桌”建模成两个完全不同的工作区，并让观战、坐下、准备、对局、结束和再战在同一张桌内连续演进；任何时刻只展示当前任务需要的信息。

### 核心状态流

```text
游客/账号进入
    -> 游戏大厅
    -> 进入现有牌桌或创建牌桌
    -> 观战
    -> 点击空座成为玩家
    -> 双方准备
    -> 正式对局
    -> 结束后留在原桌再战，或返回大厅
```

朋友邀请是旁路：

```text
邀请指定用户/告知桌号
    -> 进入指定牌桌
    -> 观战或坐下
```

### 入口逻辑

- “登录”和“游客”并列，游客一键进入，不先要求填写昵称或理解账号体系。
- 游客身份由产品自动提供；账号是统计、联系人和长期身份的增强层，不阻塞首次游戏。
- 桌面端使用独立游戏窗口隔离游戏客户端；移动设备复用当前窗口。
- 可借鉴的是“进入在线游戏后立刻进入可操作空间”，不是独立弹窗这一旧实现。

### 大厅逻辑

PlayOK 大厅的核心任务是扫视、判断并选择一张桌，而不是展示个人资料或解释功能。

- 主区域是活动桌列表。
- 每一行同时给出桌号、计时规则、玩家昵称/等级、空座或进行中状态以及进入动作。
- 已有空座的桌可直接加入；对局中的桌可进入观看。
- 创建牌桌是大厅级动作；高级规则在创建后、开局前配置。
- 桌面宽屏下，公开客户端使用约 71% 的桌列表区域和 29% 的在线用户区域；两边各自滚动，不拉长整个页面。
- 大厅聊天和在线用户属于大厅社交层；它们不会跟随用户进入正式对局后继续占据主区域。

### 牌桌逻辑

- 进入牌桌默认可以是观战者，用户点击空座后才成为玩家；“进入桌子”和“参加比赛”不是同一个动作。
- 玩家位是明确的空间对象：空座、已坐下、已准备、已开始都有直接可见的状态。
- 双方确认开始后进入正式对局，不需要重新导航或创建另一个房间。
- 对局结束后仍停留在同一张桌，适合连续再战和保留对手关系。

### 正式对局布局逻辑

```text
┌────────────────────────┬─────────────────┐
│                        │ 桌号 / 计时规则  │
│                        │ 黑白玩家 / 时钟  │
│         棋盘           ├─────────────────┤
│                        │ 聊天 历史 用户 设置│
│                        │ 当前标签内容     │
└────────────────────────┴─────────────────┘
```

- 棋盘始终是最大、最稳定的交互对象。
- 右侧固定面板先显示桌号、双方玩家、等级/时钟和当前状态。
- 聊天、历史、用户、设置共用同一块标签区域；切换标签只替换内容，不改变棋盘位置。
- 和棋、悔棋等桌内动作只在相关状态出现或可用，不把大厅动作带入对局。
- 桌面端公开客户端使用固定约 310px 的右侧栏；窄屏改成可收起/切换的侧栏，而不是让所有内容纵向同时展开。
- 棋谱历史与前后步回放在桌内即可查看，不需要离开当前棋盘。

### 用户操作习惯

PlayOK 培养的是一套稳定动作语法：

1. 进入大厅先扫桌子，而不是先编辑资料。
2. 点击桌子先观看现场，再决定是否坐下。
3. 点击空座表示“我要参加”；准备表示“我可以开始”。
4. 开局后注意力固定在棋盘和玩家时钟，聊天/棋谱/设置按需切换。
5. 一局结束仍留在原桌，默认期待是再来一局，而不是重新经历匹配流程。

### 本项目应采用

- 把当前在线模式拆成 `OnlineLobbyView` 和 `GameTableView`，由是否已进入房间决定，两者不能纵向同时显示。
- 保留一键游客身份；昵称编辑和注册移动到账号层，不作为进入大厅的必填步骤。
- 大厅主区域改为可扫视的桌子列表，每行直接展示加入决策所需信息。
- 区分“进入/观战”和“坐下成为玩家”；观战者可在牌桌内点击空座补位。
- 正式牌桌固定为“棋盘 + 桌内侧栏”；侧栏用标签复用空间。
- 牌桌动作由状态驱动：观战时坐下，等待时准备，进行时悔棋/认输，结束时再战/离开。
- 邀请链接直接进入指定牌桌，保留本项目比手动找桌号更顺畅的优势。
- 对局结束后保留牌桌与对手，优先支持连续再战。

### 不应照搬

- 不复制 PlayOK 的独立弹窗、小字号、旧式图标和视觉样式。
- 不复制其较繁琐的朋友查找/邀请流程。
- 不把全部用户、全部聊天和设置永久放在手机屏幕上；移动端应使用棋盘优先的标签或底部抽屉。
- 不复制客户端代码、CSS、截图素材或文案。

### 可执行设计任务

1. 定义在线视图状态：`lobby / table_spectating / table_waiting / table_ready / table_playing / table_finished`。
2. 将当前同时承担大厅和桌内职责的 `FriendRoomControls` 拆成大厅视图和牌桌视图。
3. 设计桌子列表的单行信息模型与桌面/手机布局。
4. 设计固定桌内侧栏及 `状态 / 聊天 / 棋谱 / 用户 / 设置` 标签。
5. 定义每个状态允许出现的最多 3～4 个主要动作。
6. 验证邀请链接能绕过大厅直达指定牌桌，并正确恢复游客或账号身份。

### 验收问题

- 用户能否不填写表单直接看到可加入的桌子？
- 用户能否一眼区分观战、空座、已坐下、已准备和进行中？
- 进入牌桌后，大厅列表和排行榜是否完全退出主视野？
- 切换聊天/棋谱/用户/设置时，棋盘是否保持尺寸和位置？
- 一局结束后能否在原桌直接再战？
- 手机端能否在第一屏附近看到当前状态和完整棋盘？

### 关键证据

- `client/src/components/Join.js:11-85`：加入和创建使用同一张入口卡片，通过本地状态互斥切换。
- `client/src/components/Lobby.js:20-172`：创建/加入房间后进入等待界面，双方到齐才开放 Start；开始后等待界面整体替换为游戏界面。
- `client/src/components/Game.js:56-119`：处理对手离开、终局和当前回合；结束后的主路径是回到首页重新加入新游戏。
- `client/src/components/Join.js:12-45`：小于 480px 的旧实现直接显示“不支持手机”的说明。
- 在线演示在 2026-07-10 仍能打开，首屏只显示昵称、房间码、Join Game 和次级 Create 入口。

### 一句话结论

这个项目的强项是把好友对局压缩成“加入/创建 -> 等待双方 -> 开始 -> 下棋”四个极清楚的界面状态；弱项是把每局当作一次性会话，缺少大厅、观战、再战和移动端连续体验。

### 入口逻辑

- 默认入口是加入已有游戏：昵称、房间码、Join Game。
- “没有房间码？”下方的 Create 链接切换成创建模式；创建模式只要求昵称，房间码由服务端生成。
- Join 与 Create 互斥显示，用户不会同时面对两套表单和多个同权按钮。
- 与 PlayOK 相比，它更适合“我已经有一个朋友和房间码”的一次性邀请场景，不适合浏览公共桌子。

### 等待房逻辑

- 进入后立刻显示房间码，并提供点击复制。
- 黑方和白方各有独立座位卡；空位直接显示等待对手。
- 第二位玩家进入后，Start 从 disabled 变为可用。
- 任意玩家都可以触发开始；开始事件广播后，所有客户端同时把等待界面替换成棋盘。
- 对手在开局前离开时，房主仍停留在等待房，Start 再次不可用。

### 正式对局逻辑

- 进入 playing 状态后，等待房内容整体隐藏，不继续占用棋盘周围空间。
- 棋盘上方只保留当前回合、玩家名和颜色。
- 非当前玩家点击、已有棋子位置点击、终局后点击使用即时提示，不改变界面结构。
- 对手中途离开直接进入结果提示。
- 终局动作是 `Join a New Game` 返回首页，而不是留在原房连续再战。

### 用户操作习惯

```text
有邀请码：填昵称和房间码 -> 加入 -> 等待/开始 -> 下棋 -> 回首页
无邀请码：切换创建 -> 填昵称 -> 获得房间码 -> 分享 -> 等待 -> 下棋
```

用户始终只处理当前阶段的一件事，这是该项目最值得借鉴的部分。

### 本项目应采用

- 创建和加入应是两个清楚的子流程，不长期并排展示两个输入框和三个同权动作。
- 创建成功后直接进入明确的“等待桌”状态，突出邀请码、两个座位和下一步。
- 从等待到开始应整体切换界面；正式对局不能继续显示大厅资料、公共聊天、排行榜和房间列表。
- 等待对手、对手加入、可以开始、对手离开都应有可见且即时的状态反馈。
- 可以保留本项目已有的邀请 URL，进一步省掉手动填写房间码。

### 不应照搬

- 不采用“一局结束就回首页”的一次性会话；应保留原桌、对手和再战路径。
- 不让任意一方单方面开始；本项目继续使用双方准备或服务器明确的开始条件。
- 不复制其手机端阻断策略；本项目必须让邀请、等待和对局在手机上完整可用。
- 不复制满屏动画背景、弹窗和旧 Bootstrap 视觉。
- 无 LICENSE，不复制其组件、Socket 代码、样式和截图。

### 对本项目的补充启发

PlayOK 适合公共大厅和长期牌桌模型；scheng20 更擅长朋友邀请的最短路径。两者可以组合：

```text
公共用户：在线大厅 -> 选桌/建桌 -> 牌桌
朋友邀请：邀请 URL -> 直接进入等待牌桌
```

无论从哪条路径进入，一旦进入牌桌，就使用统一的观战、坐下、准备、对局和再战状态机。

### 来源锁定

- 访问日期：2026-07-10

### 关键证据

- `client/src/App.js:19-32`：大厅、排行榜、本地游戏、说明和在线牌桌是独立路由。
- `client/src/Components/Navbar.js:87-112,195-225`：桌面和移动导航都把 Lobby、Leaderboard、Info、Local Play 分开。
- `client/src/Components/Lobby/Room/RoomForm.js:54-63,142-168`：Find a Game 优先加入第一个开放公共桌，无桌时自动创建；Custom Game 是独立入口。

- `client/src/Components/GameRoom/GameRoom.js:64-109,237-248`：牌桌使用两列布局；左侧是房名、玩家和评分变化，右侧在两人到齐前显示等待，到齐后显示棋盘。
- `client/src/Components/GameRoom/GameBoard.js:68-106`：对局区只显示对手离开、当前回合/胜者和棋盘。
- `client/src/Components/BlockingForm/LeaveForm.js:35-105`：等待阶段离开前拦截导航并明确说明房间会被删除。
- 在线演示在 2026-07-10 仍可访问；游客可以浏览 Lobby 和独立 Leaderboard，但首页明确提示登录后才能参加在线比赛。

### 一句话结论

这个项目最值得借鉴的是“功能分区”和“双通道开局”：想立即玩的人点 Find a Game，想控制规则或邀请朋友的人点 Create Custom Game；房间列表和排行榜各自承担单一任务，不挤进棋盘页面。

### 全局信息架构

```text
全局导航
├── Lobby
├── Leaderboard
├── Info
├── Local Play
└── Login / Account
```

- 游客仍可以浏览大厅和排行榜，先理解社区活跃度和排名结构。
- 在线参赛需要账号，权限边界清楚，但首次开局成本高于 PlayOK。
- 排行榜不作为大厅里的一个超长模块，而是独立路由和独立搜索表格。
- 本地游戏和在线大厅是不同任务空间，不在同一棋盘页上叠加控件。

### 大厅双通道

大厅顶部对登录用户提供两个并列但意图不同的入口：

1. `Find a Game`
   - 查找第一个无密码且未满的桌子。
   - 找到就直接加入并进入 `/play/<room>`。
   - 没找到则自动生成房名并创建新桌。
   - 用户不需要理解房间码、房间列表或设置即可开始。
2. `Create Custom Game`
   - 打开模态表单。
   - 设置自定义房名和可选密码。
   - 适合朋友对局或需要私密性的用户。

这是比“随机匹配、创建、加入三个同权按钮”更符合用户意图的分组：快速开始是一条完整默认路径，自定义创建才暴露更多设置。

### 房间列表逻辑

- 列表位于快速动作之后，用于有明确选择需求的用户。
- 搜索只针对房名，降低浏览成本。
- 单行直接展示房名、玩家对阵、人数 `x/2`、是否需要密码和当前动作。
- 未满显示 Join；已满显示 In Game。
- 密码输入仅在相应房间行中出现，不让所有用户提前处理无关字段。
- 不支持观战是明显缺口：满桌只显示 In Game，用户无法像 PlayOK 一样进入学习或等待补位。

### 牌桌与对局逻辑

- 进入牌桌后使用两列：玩家/房间信息与棋盘。
- 左侧展示房间名、双方头像/等级和胜负积分变化，右侧是当前任务。
- 只有一人时右侧只显示 Waiting for others；第二人进入后直接出现棋盘。
- 没有显式“坐下”和“准备”阶段，加入房间就占用玩家位，两人到齐即进入可下状态。
- 对局中只显示 Current Turn 或 Winner，保持棋盘区简洁。
- 离开等待房会先提示后果；对局中离开会触发惩罚和对手通知。
- 检查范围内未看到清楚的原桌再战入口，这一点弱于 PlayOK 的长期牌桌模型。

### 用户操作习惯

```text
只想马上玩：登录 -> Find a Game -> 自动加入/创建 -> 等待或下棋
想找特定桌：登录 -> 浏览/搜索房间 -> 必要时输入该桌密码 -> Join
想和朋友玩：登录 -> Create Custom Game -> 房名/密码 -> 分享房名 -> 等待
想看排名：全局 Leaderboard -> 搜索用户
```

### 本项目应采用

- 在线大厅提供一个真正的一步式 `快速匹配`，其内部决定加入开放桌还是进入匹配队列；用户不处理房间码。
- 把 `创建朋友桌` 作为第二条路径，高级设置按需展开，而不是常驻主界面。
- 房间列表每行只显示加入判断所需的信息，密码输入和其他字段按行/按需出现。
- 排行榜、个人资料和本地模式从大厅与正式牌桌中分离，使用全局导航或独立视图。
- 离开等待桌或进行中对局时明确说明后果，防止误退。
- 牌桌侧栏可展示双方评分和本局可能的积分变化，但不能挤占棋盘。

### 不应照搬

- 不强制注册后才能在线玩；本项目保留游客即时参与，注册只增强长期统计和社交。
- 不使用“加入第一个数组元素”作为真实匹配策略；应由服务端按状态、规则和未来评分条件匹配。
- 不让两人到齐就无提示开局；本项目继续保留明确准备状态。
- 不把满桌变成不可进入的 In Game；应允许观战。
- 不复制客户端持有/修改房间数组的信任模型；本项目保持服务端权威。
- 无 LICENSE，不复制其 React 组件、Tailwind 组合、数据结构或素材。

### 对本项目的补充启发

minh100 的最佳贡献不是具体页面样式，而是用用户意图划分入口：

```text
立即开始 -> 快速匹配
和朋友玩 -> 创建朋友桌 / 邀请链接
有目标地找桌 -> 桌子列表
查看长期数据 -> 排行榜 / 资料独立视图
```

这四种意图不应继续混在同一块 `FriendRoomControls` 中。

- `src/main.js:183-213`：回合、AI 思考、胜负和自动滚动棋谱分别反馈。

### 一句话结论

sen 的界面不是“开始页 + 游戏页”，而是一张始终可操作的棋盘工作台：设置、动作、状态和棋谱在固定窄侧栏内连续更新，棋盘始终是视觉与操作中心。

### 页面与状态逻辑

```text
打开页面
    -> 默认以上级难度、执黑先手直接进入可下状态
    -> 用户落子
    -> AI 思考提示
    -> AI 落子
    -> 回到用户回合
    -> 胜/负/和棋
    -> 原地新局或悔棋后继续
```

它没有额外的“确认设置并开始”门槛。难度、先后手、新局、悔棋、当前状态和棋谱始终在棋盘旁边；用户到达页面即可落子。

### 信息层级

- 页头只承担产品名、语言和主题等低频全局动作。
- 左侧栏从上到下按任务顺序排列：开局参数、主要动作、当前状态、历史记录。
- 棋盘不是卡片中的一个小模块，而是主区，自动取得侧栏之外的全部宽度。
- 状态文案同时说明“轮到谁”和“AI 是否在计算”，不用用户从棋盘变化反推系统状态。
- 棋谱自动滚到最后一步，并对最后一步加视觉强调；历史是辅助信息，不与棋盘争抢主区。
- 移动端仍保留全部能力，但把侧栏改为棋盘上方的紧凑工具区，并限制棋谱高度。

### 控件行为

- 默认配置已经能玩，降低首次进入成本。
- 切换执黑/执白会立刻开始一局新棋；该动作有强副作用，却没有二次确认。
- 切换难度不会打断当前棋局，新难度从下一次 AI 计算起生效。
- 悔棋以“用户一步 + AI 一步”为一个语义单位，而不是机械撤销单个棋子。
- 即使已经判胜，也允许悔棋回到对局，适合练习场景。
- 切换语言和主题不会改变棋局状态。

### 用户操作习惯

```text
立即练习：打开页面 -> 直接落子
更换先后手：选择颜色 -> 自动新局 -> 继续落子
调整强度：选择难度 -> 当前局继续
纠正一回合：待った/Undo -> 撤回双方最近一轮 -> 继续
复盘当前局：在棋盘旁滚动查看棋谱
```

### 本项目应采用

- 本地双人和 AI 模式直接进入棋盘工作区，不经过在线大厅，也不要求再点一次“创建房间”。
- 为 AI 模式提供一个可直接玩的默认配置；高级参数可修改，但不能阻塞首次落子。
- 桌面端把低频设置、主要动作、回合状态和历史放入同一固定侧栏；棋盘取得最大连续空间。
- AI 计算期间必须明确显示“AI 思考中”，并暂时阻止误操作。
- AI 悔棋按完整回合撤回；在线对局的悔棋则应是双方协商动作，两者不能共用同一语义。
- 移动端将侧栏压到棋盘下方或上方的可折叠区，默认只保留当前回合和必要动作。
- 终局后保持棋盘和历史可见，让“再来一局”“复盘”“返回”成为同一工作区里的下一步。

### 不应照搬

- 不让修改先后手静默清空当前棋局；有进度时应提示，或把设置锁到下一局。
- 不让全部历史长期占用移动端首屏；默认折叠或仅显示最近一步。
- 不把本地模式的持续侧栏直接套用到在线大厅；大厅和牌桌仍应是独立页面状态。
- 不将语言和主题提升为对局中的高优先动作，它们只适合页头或更多菜单。

### 对本项目的补充启发

PlayOK 解决的是“多人牌桌如何连续存在”，sen 解决的是“棋盘工作区如何连续存在”。两者可以组合成同一原则：

```text
大厅负责选择桌子，不放棋盘工具
牌桌负责持续对局，不返回创建表单
本地/AI 负责直接练习，不进入在线大厅
进入任何工作区后，状态在原地演进，主任务的位置不跳动
```

### 一句话结论

gkoos 最值得借鉴的不是单列样式，而是它把“配置普通对局”“摆设棋盘”“正式落子”视为互斥状态：用户每一步只看到当前任务所需的控件，避免把所有功能永久堆在棋盘周围。

### 核心状态流

普通对局：

```text
未开始，棋盘禁用
    -> 选择执子和难度
    -> Start Game
    -> 参数锁定、棋盘启用、显示当前回合
    -> AI 回合显示进度
    -> 终局显示结果
    -> New Game 回到开局状态
```

摆设对局：

```text
未开始
    -> Setup Board
    -> 普通参数被替换为电脑执子、下一手、开始、取消
    -> 点击格点依次切换 空/黑/白
    -> Clear Board / Cancel / Start Game
    -> 从指定局面进入正式对局
```

### 交互逻辑

- 棋盘从一开始就在页面上，但未开始时不可落子，状态文案解释下一步。
- “开始”是模式边界：点击后参数锁定，避免中途改变规则导致棋局语义不一致。
- 新游戏不是额外常驻按钮，而是在正式对局阶段替换 Start Game 的位置。
- 摆设不是一个悬浮面板叠加在普通设置上，而是切换整个工具区的内容。
- Setup Board 的退出路径清楚：开始指定局面或取消返回普通开局。
- AI 回合不仅写“思考中”，还显示计算进度；用户知道系统没有卡死。
- 最后一步与获胜棋子都有专门标记，状态反馈同时落在文案和棋盘上。

### 响应式策略

- 整体保持单列，工具区在棋盘上方自动换行。
- 棋盘始终保持 1:1，并随 375、480、768、1024、1200px 断点增加上限。
- 桌面端仍考虑低高度窗口：视口低于 900px 时主动缩小棋盘和间距，避免必须滚动才能看到整盘。
- 移动端控件字号和间距缩小，但可用动作不消失。
- 这种策略适合“单棋盘 + 少量控制”的本地模式；在线牌桌还需要容纳玩家、准备、聊天和观战，不能直接照搬单列。

### 用户操作习惯

```text
普通人机：选执子/难度 -> Start Game -> 落子 -> New Game
残局练习：Setup Board -> 摆子 -> 指定电脑颜色/下一手 -> Start Game
放弃摆设：Setup Board -> Cancel -> 返回普通设置
等待 AI：查看 AI Thinking 和进度条，不重复点击棋盘
```

### 本项目应采用

- 把开局参数当作一个明确的“开局前”状态；正式对局后锁定会改变规则的参数。
- 同一位置随状态替换主动作：`开始` → `进行中必要动作` → `再来一局`，避免按钮越积越多。
- 高级的摆局/分析能力放到独立模式，不在普通对局侧栏中永久展示。
- 每个模式都必须有清楚的进入、取消和完成路径。
- 棋盘尺寸同时受可用宽度和可用高度约束，桌面低高度窗口也要完整可玩。
- AI 的异步状态要有明确反馈；是否显示精确百分比取决于计算是否能提供真实进度，不能用虚假进度装饰。

### 不应照搬

- 本项目 AI 模式已有可直接开玩的价值，不必一律禁止棋盘直至点击 Start；可以用默认配置直接开始，并把显式开始保留给自定义/摆局模式。
- 不应只靠 `visibility: hidden` 留下不可见按钮占位；组件应按状态真正切换，同时保持布局稳定。
- 不把 AI 思考进度伪装成精确完成度；无法度量时使用活动指示和状态文案。
- 摆局开始前要校验黑白子数量、已成五等非法局面；该项目源码目前接受任意局面。
- 无 LICENSE，不复制其实现。

sen 与 gkoos 分别代表两种合理入口，适用于不同用户意图：

```text
默认练习：直接进入可下棋盘（sen）
自定义练习/摆局：配置 -> 明确开始 -> 锁定规则（gkoos）
```

本项目不必二选一。AI 首页可以默认直接可玩；当用户进入“自定义”或“摆局”时，再切换到有明确开始边界的配置状态。

### 一句话结论

BGA 的核心不是“平台统一用了什么按钮颜色”，而是建立了一套稳定的桌面语法：玩家先看到当前状态，再直接操作棋盘；只有无法在棋盘上表达的特殊动作进入动作栏；身份信息紧凑地留在桌边，历史、规则和设置下沉为随时可访问的次要层。


### 当前动作逻辑

- 普通落子直接发生在棋盘交叉点，动作栏不重复提供“落子”按钮。
- 当前变体需要选择颜色时，动作栏显示清楚的任务说明和 `Swap Colors` 等特殊动作。
- 动作栏始终同时回答两个问题：现在是什么状态；我现在能做什么。
- 官方指南要求主推进动作居中，取消/撤销靠右并与推进动作分隔。
- 动作栏上限为“上下文 + 最多四个按钮”，保证移动端不会溢出。
- 只在“动作存在但此刻不可用”时禁用；当前上下文永远无关的动作应直接隐藏。
- 对不可逆且影响大的操作使用确认，但避免每一步都叠加确认、撤销和重置。

### 对局反馈与连续性

- 当前状态、合法目标高亮和落子结果形成连续反馈，不让用户猜下一步。
- 最后一步可以按用户偏好高亮，棋盘坐标也可开关；设置被归入统一选项入口。
- 每个关键动作进入日志，说明谁在什么坐标落子、谁换色、谁取消、谁获胜。
- 回放按步推进，状态文字、棋盘高亮与日志保持同步。
- 刷新或断线后要恢复同一棋局、同一当前动作和临时高亮；服务端是关键状态的唯一事实源。
- 终局仍保留完整棋盘、结果与回放入口，历史不是一次性提示。

### 移动端原则

- 主棋盘和当前动作仍在前面；玩家面板从右侧重排到顶部，日志和其他次要区下沉。
- 游戏核心应在 100% 缩放下可操作，不依赖用户手动缩放才能看清。
- 交互目标至少 32×32px，理想值 40–44px；空间不够时使用滚动或次级面板，不继续缩小。
- 棋盘周围保留无动作留白，既减少误触，也允许用户在移动端滚动页面。
- 不使用阻塞式弹窗打断对局；教程弹窗也必须可快速跳过。

### 用户操作习惯

```text
不了解游戏：产品页 -> Example game / Rules -> 观察回放
想立即玩：Play -> 进入平台匹配/建桌流程
想观战：Watch a game in progress -> 选择桌子
正式对局：看顶部当前动作 -> 直接点击棋盘 -> 等待对手
特殊开局：看动作栏说明 -> 执行换色/确认等非棋盘动作
复盘：棋盘下方日志或独立回放 -> 逐步前进/后退
```

### 本项目应采用

- 牌桌顶部建立单一“当前任务栏”，明确显示轮到谁、等待谁以及当前必要动作。
- 落子继续直接在棋盘完成；准备、确认、和棋回应、认输、再战回应等非棋盘动作进入任务栏。
- 同时可见的任务栏按钮控制在四个以内；隐藏无关动作，不把所有状态按钮永久禁用排列。
- 玩家信息区只放桌边判断所需内容，不混入规则设置、房间创建或排行榜。
- 聊天、用户、历史、规则和设置使用侧栏页签或移动端折叠区，保持随时可达但不抢棋盘。

### 不应照搬

- 官方历史年度报告曾公开 92,000 名独立访客、6,324 名新增注册、9,844 场锦标赛；Gomoku/Renju 当年达到 8,178 玩家小时。该数据较旧，只作为长期规模证据，不当作当前月活。
- 2025 Gomoku 系列赛连续 30 天每天举办一场 Swap2 七轮赛，并汇总最佳 20 场成绩。
- 2025/26 国际学生赛事由 RIF、Vint.ee 和 Estonian Renju Union 组织，2026 年仍有赛程和结果。
- 历史学生单场 Gomoku 赛事曾达到 161 人，说明其赛事能力经过真实群体使用。

### 一句话结论

Vint.ee 是“竞技俱乐部式”产品：访客可以进入房间观察真实用户、对局和聊天，但正式建桌、邀请和赛事要求稳定身份；普通对局、电脑练习、排名、连玩奖励、锦标赛和训练被组织在同一游戏社区内。

### 全局信息架构

首页不是单一 Gomoku 落地页，而是完整平台入口：

- Games：进入各游戏房间。
- Leaderboards：跨游戏与单游戏评分。
- Forum：长期社区讨论。
- Events calendar：当前活动。
- For trainers：教师/教练组织能力。
- Mind sports olympiad：学校赛事体系。
- Game rules、Statistics：规则和公开数据。

Gomoku Top 5 在首页直接展示，竞技身份在进入游戏室前已经可见。

### 游戏室逻辑

进入 `/games/#gomoku` 后，访客自动获得 `Visitor <number>` 临时身份，不先弹注册框。游戏室首屏包含：

```text
房间层：Open rooms、Lobby、Gomoku
内容页签：Users、Chat、Games，并显示实时数量
主要动作：Create table、Send invitation、Play with computer
个人状态：I am busy
社区信息：Streaks、Best streaks、Rankings
对局列表：进行中的比赛、Watch、Interesting games
```

- 用户列表直接显示昵称和评分，先判断对手再发邀请。
- Send invitation 在未选择用户时禁用，属于上下文动作。
- Games 显示当前进行中的对局；选择对局后才能 Watch。
- 房间聊天同时给出可分享的 Gomoku 房间 URL 和复制入口。
- `I am busy` 让用户明确退出被挑战状态，而不是靠离开页面或忽略请求表达。
- 当前房间内还展示单游戏评分纪录和连玩天数，强化长期社区感。

### 游客与注册边界

- 游客可以进入游戏室、看到玩家/对局数量、评分、排名、聊天信息和观战结构。
- 实测游客点击 Create table 后，系统在房间消息中反馈 `This functionality is only for registered users`。
- 正式建桌、主动邀请和参加系列赛需要注册；2025 系列赛还要求 verified user。
- 相比 PaperGames，这里把注册放在“实际参与竞技”的边界，而不是“了解社区”的边界。
- 这种边界适合强调赛事可信度，但会增加首次在线落子的成本。

### 锦标赛、系列赛和训练

Vint.ee 将竞争强度分成多个层次，而不是把全部内容叫“排行榜”：

```text
普通游戏室 -> 自由建桌/邀请/电脑练习
Tournament -> 某个时间进入房间，完成一场正式赛
Tournament series -> 多天多场，累计最佳若干场成绩
Trainings -> 面向规则或赛事准备的练习房
Mind sports olympiad -> 学校、年龄组、团队和个人综合赛事
```

- 2025 系列赛每天固定时间开始，Swap2、7 轮、5 分钟 + 每步 2 秒。
- 30 场中计算最佳 20 场，允许错过部分日期，又奖励持续参与。
- 学生赛事使用 11/13 轮瑞士制、3 分钟 + 每步 2 秒，并配置 Buchholz 等破同分规则。
- 赛事房提前开放；用户先进入指定房间，再注册或等待系统配对。

### 长期激励

- Streak 定义为连续有玩该游戏的天数，每 100 天奖励站内 credits。
- 评分、单项纪录、连续天数和系列赛成绩是不同维度，不用一个总积分解释所有成长。
- 首页、房间、排行榜、个人档案和赛事表相互链接，用户能从当下对局追溯到长期记录。
- Credits 可连接 VIP 或站内商店，但不改变基本棋局规则。

### 用户操作习惯

```text
先观察：直接进入 Gomoku 房 -> 看 Users/Games/评分 -> 决定是否注册
找对手：选择用户 -> Send invitation -> 建桌/接受设置 -> 对局
开公开桌：Create table -> 设置规则/计时 -> 等待挑战
先练习：Play with computer -> 进入 AI 对局
观战：Games -> 选择进行中比赛 -> Watch
拒绝打扰：勾选 I am busy -> 留在房间但不接邀请
参加赛事：Tournaments/Series -> 到点进入房间 -> 注册 -> 系统轮次配对
```

### 本项目应采用

- 大厅页签直接显示用户数、开放桌数、进行中桌数，让社区状态可扫视。
- 选择用户后再出现“挑战/邀请”，选择对局后再启用“观战”，贯彻上下文动作。
- 增加明确的“暂不接受挑战/忙碌”状态，而不是让用户反复拒绝邀请。
- 游戏房链接保持稳定，朋友可以先到同一房间，再挑战或进入指定桌。
- 将普通对局、训练、单场赛事和系列赛视为层级递进的竞技路径，共用账号、评分和牌桌。
- 长期激励拆成 rating、活跃 streak、赛事成绩等独立维度，避免单一积分失真。
- 赛事允许“最佳 N 场计入总分”，降低长系列对缺席用户的惩罚。

### 不应照搬

- 不要求注册后才能完成第一局普通在线体验；本项目继续允许游客建/加入普通桌，只有正式 rating 和赛事需要稳定身份。
- 不把 Shop、VIP、论坛、教练功能、奥林匹克和统计全部放入移动端游戏首屏。
- 不让 Create table 对游客看起来可用，点击后才在聊天流中报错；应提前解释身份要求，或允许游客创建非计分桌。
- 不用很小的页签和密集列表承载移动端核心路径；用户、对局、聊天需要清晰的分段与触摸尺寸。
- 不在产品规模尚小时提前复制完整赛事后台；先保证房间、牌桌、评分和结构化记录可复用。
- 无 LICENSE，不复制客户端实现、样式、图标或赛事页面代码。

### 对本项目的补充启发

Vint.ee 展示了社区成熟后的合理分层：

```text
游客了解活跃度 -> 普通对局建立习惯 -> rating 建立身份
-> 日常训练 -> 单场赛事 -> 长系列/学校赛事
```

当前项目不需要立刻实现最后两层，但房间、历史和评分数据必须避免一次性设计，否则未来赛事会被迫建立第二套牌桌系统。

## 综合设计原则

### 1. 先按用户意图分流，再设计页面

```text
在线公开对局 -> 在线大厅 -> 牌桌
和朋友玩   -> 创建朋友桌/邀请链接 -> 指定牌桌
本地双人   -> 直接进入本地棋盘
挑战 AI    -> 直接进入默认 AI 棋盘
观战       -> 大厅/进行中列表 -> 牌桌观战位
复盘       -> 历史/回放 -> 只读牌桌
```

当前项目最需要消除的是：把快速匹配、创建、加入、AI、本地、房间设置和正式对局同时放在一个页面层级。

### 2. 大厅与牌桌是两个工作区

- 大厅回答“去哪张桌、和谁玩、现在能不能加入”。
- 牌桌回答“我在这张桌是什么身份、当前该做什么、棋局发生了什么”。
- 进入牌桌后，不再显示大厅的创建/加入表单。
- 一局结束后默认留在牌桌，支持再战、复盘或退出，不强制回到入口。

### 3. 牌桌用显式状态机驱动界面

```text
spectating
  -> seated
  -> ready / waiting-opponent-ready
  -> playing-my-turn / playing-opponent-turn
  -> finished
  -> rematch-ready / leaving
```

每个状态只渲染当前相关的主要动作。动作位置稳定，文案和可用性随状态变化。

### 4. 棋盘周围只保留三层信息

1. 当前任务：轮到谁、等待谁、需要准备/确认/回应什么。
2. 桌边身份：双方玩家、座位、颜色、计时和必要评分。
3. 次要工具：聊天、历史、用户、规则和设置，通过页签/折叠访问。

创建房间、排行榜、个人资料、全局导航不属于这三层。

### 5. 状态替换控件，不让功能不断堆叠

```text
空座时：坐下
已坐下：准备 / 离座
已准备：取消准备或等待
我的回合：在棋盘落子 + 少量协商动作
对手回合：等待状态 + 少量协商动作
终局：再来一局 / 复盘 / 退出
```

同一阶段不应同时显示“加入、创建、准备、开始、再战”等互斥动作。

### 6. 桌面和移动端保持相同优先级，而非相同排版

- 桌面：棋盘居中或占主区；固定侧栏承载桌边与页签。
- 移动端：当前任务 → 棋盘 → 必要主动作优先；玩家与次要信息重排为紧凑条/折叠区。
- 棋盘同时受宽度和高度约束，低高度桌面窗口也要完整可操作。
- 所有高频目标按触摸设计，不能只依赖 hover。

### 7. 大厅必须展示真实活跃度，但不能制造空场感

- 活跃社区可以像 FlyOrDie/Vint.ee 一样展示在线用户、进行中比赛和可观战对象。
- 冷启动或低峰期优先提供 PaperGames 式快速匹配和朋友链接，不让用户先面对空桌列表。
- 在线人数、开放桌、进行中桌和预计等待是不同信号，不能用一个含糊的“在线”数字替代。
- 数据必须来自服务端实时状态，不使用虚假热度或静态占位。
- 排行榜和赛事可以证明社区长期存在，但不能取代“现在能否开局”的即时反馈。

### 8. 身份和竞技深度按参与阶段递进

```text
游客：立即玩普通桌、浏览活跃度、观战
临时昵称：朋友链接、随机休闲对局
注册账号：长期 rating、好友、统计和跨设备恢复
验证账号：正式赛事、奖金/奖励和反作弊约束
```

- 注册应增强长期价值，不阻塞第一局普通对战。
- 只有需要可信身份的功能才提升门槛，并在点击前解释原因。
- 日榜、rating、streak、系列赛成绩分别表达短期活跃、竞技水平、持续参与和正式成绩，不能混成一个积分。
- 赛事是普通牌桌之上的配对与记分层，不是第二套棋盘客户端。

## 对当前项目的实施建议

按下面顺序实施，避免只移动按钮但不改变页面逻辑：

1. 定义 `OnlineLobbyView`、`GameTableView`、`LocalGameView`、`AiGameView` 四个互斥工作区。
2. 将当前 `FriendRoomControls` 拆成大厅动作、朋友桌创建和牌桌内动作三个边界。
3. 为牌桌建立统一任务栏，并由观战/座位/准备/回合/终局状态机提供动作。
4. 桌面端建立固定桌边栏；移动端把同一内容改为紧凑状态条和可折叠页签。
5. 让正式对局、终局和再战都留在同一牌桌上下文，只有主动退出才回大厅。
6. 将历史改为结构化事件，为刷新恢复和回放保留一致数据。
7. 大厅顶部增加真实的在线用户、开放桌、进行中桌和快速匹配状态；低活跃时优先展示 AI/朋友链接，不展示空列表主屏。
8. 用户/比赛列表使用上下文动作：选用户后挑战，选比赛后观战，并提供“忙碌/不接受挑战”状态。
9. 将多局、计时、规则和是否计分建模为 `matchConfig`，创建后锁定并在牌桌中只读展示。
10. 为未来日赛/系列赛预留 tournament、round、pairing、score 关系，但暂不实现完整赛事后台。

## 交互验收清单

- 新用户能否在不理解“房间码”的情况下开始一局？
- 与朋友对局能否通过一个链接进入正确牌桌？
- 进入牌桌后是否完全看不到创建/加入房间表单？
- 观战者是否能一眼看见空座、坐下入口和双方准备状态？
- 任一状态下是否只有一个明确的主要下一步？
- 落子是否直接在棋盘完成，而不是绕到外围按钮？
- 结束后是否保留棋盘、玩家、聊天和历史，并可在原桌再战？
- 刷新后是否回到完全相同的身份、棋局、最后一步和待处理动作？
- 390px 宽屏幕是否仍先看到当前任务和可操作棋盘？
- 低高度桌面窗口是否无需滚动即可完成落子？
- 触摸目标是否至少 32×32px，且关键目标接近 40–44px？
- 无关动作是否被隐藏，而不是作为一排禁用按钮长期存在？
- 大厅是否分别显示在线用户、开放桌和进行中桌，而不是一个无法解释的总数？
- 没有开放桌时，用户能否直接快速匹配、创建朋友桌或转到 AI，而不是停在空列表？
- 选择某个在线用户后，是否就地出现挑战动作；未选择时是否不占位置？
- 选择某场进行中对局后，是否出现明确观战动作？
- 游客能否完成第一局普通在线对战；需要注册的功能是否在点击前解释原因？
- 多局、计时、规则和计分设置是否在开局后锁定，并能在牌桌中随时查看？
- 日榜、rating、streak 和赛事成绩是否各自说明含义，避免用户误解？
